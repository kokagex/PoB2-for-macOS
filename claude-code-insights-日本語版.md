## 一目でわかる概要

### うまくいっていること
pob2macosグラフィックス移植プロジェクトに対して、タスク委譲を多用した数千の並列セッションを実行する、印象的に洗練されたマルチエージェントパイプラインを構築しました。Metalシェーダーレンダリングの反復的なデバッグループ（編集、ビルド、デプロイ、検証）は、真のシステムレベルの規律を示しており、カスタムスキルとルーチンへの投資は、自動化に関する賢明な長期的思考を示しています。→ [印象的な成果](#印象的な成果)

### うまくいっていないこと
Claudeの側では、間違ったファイルの修正（metal_backend.mmではなくmetal_shaders.metalを編集するなど）やバグのあるデプロイステップが、非生産的なサイクルで使用量の予算を消費しています。あなたの側では、大量の短い並列セッションが常にレート制限に突き当たっており、複数のセッションで文字通り何も達成できていません。また、ClaudeはあなたのCLAUDE.mdに、ガイダンスなしで正しいファイルとビルドステップを一貫してターゲットにするための十分なプロジェクトコンテキストが不足している可能性があります。→ [問題点](#問題が発生する箇所)

### すぐに試せること
**Hooks**を設定して、各デプロイ後にビルド成果物を自動検証する（例：正しいdylibがランタイムディレクトリに配置されていることを確認）ことで、視覚的な検証の後ではなく即座に壊れたサイクルをキャッチできます。また、CLAUDE.mdにプロジェクトのファイル構造と主要なアーキテクチャ上の決定（特にどのシェーダーファイルがどのレンダリングパスにマップされるか）を充実させることで、Claudeが最初の試行で間違った推測をするのを止められます。→ [試すべき機能](#試すべき機能)

### 野心的なワークフロー
モデルがより高性能になるにつれて、作業を個別のステップにシリアライズする**レジリエントタスクキュー**を目指しましょう。これにより、レート制限の中断が失われたセッションではなく一時停止になります。エージェントは再プロンプトなしで正確に中断した場所から再開します。さらに、あなたのMetalデバッグワークフローは**並列仮説テスト**の完璧な候補です。複数のサブエージェントが同時に異なる根本原因（シェーダーソース、ビルドパイプライン、ランタイムリンキング）を調査し、連続的な試行錯誤ではなく一回のパスで答えに収束します。→ [今後の展望](#今後の展望)

## Claude Codeの使い方

あなたはClaude Codeの**極めてヘビーで自動化駆動型のユーザー**です。わずか1週間で3,096セッション、約20,000メッセージ、2,500時間以上の計算時間を記録しています。あなたのインタラクションスタイルは**ルーチンスキルと自動化パイプライン**を中心に展開しており、macOS Metal/グラフィックスプロジェクト（pob2macos）のグラフィックスレンダリングのデバッグを中心に、事前定義されたタスクを実行するためにClaude Codeを繰り返し呼び出すシステムを構築しています。

メッセージ数に対するBash呼び出しの膨大な量（88,122回）は明確な物語を語っています：あなたは会話的なやり取りではなく、Claudeに最小限の中断で広範囲に実行させ、大規模な自動化ワークフローをオーケストレーションしています。あなたの主要な目標（グラフィックスレンダリングのデバッグ342セッション、タスク管理228、ファイル編集114）は、複雑なレンダリングパイプライン上で継続的に作業する**永続的な自動化開発アシスタント**としてClaudeを使用していることを示唆しています。

しかし、あなたの結果は重大な**摩擦が支配的な体験**を明らかにしています。セッションの大部分は、有意義な作業が開始される前にレート制限によってブロックされており、複数のセッション要約では完全に阻止されたことが記述されています。作業が進行する場合でも、**間違ったアプローチ（128件）とバグのあるコード（228件）**のパターンに遭遇し、Claudeは頻繁に間違ったファイル（例：metal_backend.mmではなくmetal_shaders.metalを編集）を修正したり、デプロイステップに失敗したりして、無駄なテストサイクルにつながっています。

分析されたセッション全体で128の結果が「部分的に達成」で、完全に達成された結果がゼロであり、114件の過剰な変更と組み合わせると、自動化ループが解決策に収束せずにスピンすることがあることを示唆しています。それにもかかわらず、あなたは**驚くほど忍耐強く**、357セッションが「おそらく満足」として登録されています。これは、あなたが段階的な進歩を受け入れ、各実行で完璧を期待するのではなく、時折コース修正が必要な疲れ知らずの働き手としてClaudeを扱っていることを示しています。

**主要パターン**: Claude Codeを大規模に並列化された自動化ファーストパイプラインとして運用し、最小限の手動介入で数千のルーチンセッションを起動し、継続的でハンズオフの開発進捗のコストとして高い摩擦と部分的な結果を許容しています。

## 問題が発生する箇所

Claude Codeの使用は、繰り返されるレート制限ブロックと誤った方向のデバッグ努力によって深刻に妨げられており、セッションの大多数は目標を部分的にしか達成していません。

### 1. セッション全体を破壊するレート制限ブロック
**問題**: APIの使用制限に繰り返し遭遇して作業が一切できず、セッション全体を無駄にしています。

**対策**: セッションの間隔を空けるか、タスクをより少ない、しかしより実質的なインタラクションに統合するか、使用ティアをアップグレードして、これらの完全損失セッションを回避することを検討してください。

**例**:
- 少なくとも5つのセッションがレート制限によって完全にブロックされ、ルーチンを実行したりプロジェクトを再開しようとして、毎回文字通り何も達成できませんでした
- 黄色テキストレンダリングを調査する1つのセッションでは、計画フェーズが完了した直後、実際の分析が開始される前にClaudeがレート制限に達し、セットアップの努力全体が無駄になりました

### 2. 無駄なサイクルにつながる間違ったアプローチ
**問題**: 作業のかなりの部分（128件の間違ったアプローチ）で、Claudeが間違ったファイルを修正したり、コース修正する前に誤った道を取ったりして、非生産的なアクションで限られた使用予算を消費しています。

**対策**: プロンプトで事前により具体的なファイルパスとコンテキストを提供することで、これを軽減できます。

**例**:
- Metalテキストレンダリングのデバッグ中、Claudeが間違ったファイル（metal_backend.mmではなくmetal_shaders.metal）を修正し、正しい場所を見つける前に複数の無駄なテスト-デプロイ-検証サイクルを引き起こしました
- スキルインストールセッションは、ClaudeがSKILL.mdファイルをコミットしたが適切に認識されず、意図したものではなく完全に間違ったスキル（/web-artifacts-builder）がトリガーされたときに横道にそれました

### 3. 過剰な変更とバグのあるデプロイパイプライン
**問題**: 228件のバグのあるコードと114件の過剰な変更が発生しており、特にエラーを複合化するビルドとデプロイステップに関連しています。

**対策**: 各デプロイ前にチェックリストまたは検証ステップを確立すること（正しいdylibがランタイムディレクトリにコピーされていることを確認するなど）により、繰り返される破損したテストサイクルを防ぐことができます。

**例**:
- 正しいdylibがランタイムディレクトリにコピーされていなかったために繰り返しデプロイが失敗し、各シェーダー修正が変更に何らかの効果があったかどうかを検証するだけで複数の試行を必要としました
- セッション全体で88,122回のBash呼び出しがあり、膨大な量の試行錯誤コマンドは、デバッグワークフローに構造化された検証ゲートが欠けており、レート制限をより速く消費する過剰な反復的変更につながっていることを示唆しています

### 既存のClaude Code機能

#### 1. Hooks（フック）
**概要**: 編集やコミットの前後など、特定のライフサイクルイベントでシェルコマンドを自動実行します。

**あなたにとっての利点**: あなたの最大の摩擦ポイントは間違ったバイナリのデプロイです。ポスト編集フックは、すべてのビルド後にdylibがランタイムディレクトリに正しくコピーされていることを自動検証でき、複数のテストサイクルを無駄にした繰り返しのデプロイ失敗を排除できます。

**設定例**:
```json
// .claude/settings.json
{
  "hooks": {
    "postToolUse": [
      {
        "matcher": "Bash",
        "pattern": "make|build|xcodebuild",
        "command": "bash -c 'echo \"ビルド検出 - デプロイを検証中...\"; ls -la /path/to/runtime/dir/*.dylib | head -5'"
      }
    ]
  }
}
```

#### 2. Custom Skills（カスタムスキル）
**概要**: 単一の/commandで実行できる、Markdownファイルとしての再利用可能なプロンプト。

**あなたにとっての利点**: すでに/routineスキルを多用しており（複数のセッションがルーチンを実行しようとする試みでした）、スキル作成は最優先目標の1つです。最も一般的なワークフロー（ビルド-テスト-デプロイ、シェーダーデバッグ）を堅牢なスキルに形式化することで、各セッションで行う反復的なセットアップを減らすことができます。

**スキル例**:
```markdown
# .claude/skills/debug-render/SKILL.md
## レンダリングパイプラインのデバッグ
1. 正しいソースファイルを特定（シェーダーパイプラインコードの場合はmetal_backend.mm）
2. 要求されたシェーダー修正を実行
3. `make`または同等のコマンドでプロジェクトをビルド
4. ビルドされたdylibのタイムスタンプを確認: `ls -la /path/to/runtime/*.dylib`
5. 必要に応じて新しいdylibをランタイムディレクトリにコピー
6. アプリを実行して視覚出力をキャプチャ
7. 前後の比較で結果を報告
```

#### 3. Headless Mode（ヘッドレスモード）
**概要**: スクリプトとCI/CDからClaudeを非対話的に実行します。

**あなたにとっての利点**: 3,096セッションがあり、多くがレート制限によって途中で停止されるルーチン/バッチタスクです。オフピーク時（例：cronジョブ）にヘッドレスモードで軽量なルーチンタスクを実行することで、レート制限の問題を回避し、ルーチンが実際に完了することを保証できます。

**使用例**:
```bash
# cronジョブやスクリプトでルーチンタスクをヘッドレスコマンドとして実行
claude -p "日次プロジェクトルーチンを実行: ビルドステータスの確認、デプロイの検証、未解決の課題の要約" --allowedTools "Bash,Read,Grep" --max-turns 20
```

### 1. レート制限セッションが使用量を消費している
**問題**: 少なくとも5-6セッションがレート制限ブロックで完全に無駄になりました。多くの短い試行ではなく、より少ない長いセッションに作業をまとめましょう。

**詳細**: データは、Claudeを開いてすぐにレート制限に達し、終了したセッション（同じことを2-3回連続で試すこともあります）が複数あることを示しています。これらの中止されたセッションは、それぞれセッション数にカウントされます（1週間で3,096セッションは非常に高いです）。作業をより少ないセッションに統合し、オフピーク時のルーチンタスクにヘッドレスモードを使用することで、無駄な試行を劇的に減らすことができます。

**Claude Codeに貼り付け**:
```
新しいセッションを開始する前に、レート制限に達する可能性があるかどうかを確認してください。その場合は、リクエストをキューに入れ、繰り返し失敗するのではなく後で再試行することを提案してください。
```

### 2. 間違ったアプローチの摩擦が最大の生産性キラー
**問題**: 128セッションで「間違ったアプローチ」がフラグされました。Claudeが間違ったファイルを修正しないように、CLAUDE.mdにアーキテクチャコンテキストを追加しましょう。

**詳細**: 摩擦データは、「間違ったアプローチ」が128件、「バグのあるコード」が228件であることを示しており、これらは他のすべての摩擦ソースを圧倒しています。シェーダーファイルの誤認識（metal_shaders.metal vs metal_backend.mm）は具体的な例です。CLAUDE.mdにファイルの目的を記載したプロジェクトアーキテクチャマップを追加することで、Claudeがどのファイルを修正するかを誤って推測するのを防ぐことができます。特に、最優先目標カテゴリ（342セッション）であるグラフィックスレンダリング作業において。

**Claude Codeに貼り付け**:
```
CLAUDE.mdに## プロジェクトアーキテクチャセクションを作成して、主要なファイルをその目的にマッピングしてください。Metalレンダリングパイプラインから始めます：どのファイルにシェーダーコードが含まれ、どのファイルにレンダリングバックエンドが含まれ、どのファイルがビルド/デプロイパイプラインを処理するか。
```

### 3. 過剰な変更パターンにはガードレールが必要
**問題**: 114セッションで「過剰な変更」の摩擦がありました。CLAUDE.mdにスコープ制約を追加しましょう。

**詳細**: 21,521回のEdit呼び出しと7,971回のWrite呼び出しで、Claudeは膨大な量の変更を行っています。114件の「過剰な変更」摩擦の出現は、Claudeが意図した以上の修正を行うことが多いことを示唆しています。「最小限の変更を行う」や「2つ以上のファイルを編集する前にスコープを確認する」といった明示的なガードレールをCLAUDE.mdに追加することで、修正を集中させることができます。特に精度が重要なグラフィックスデバッグ作業において。

**Claude Codeに貼り付け**:
```
CLAUDE.mdに追加: デバッグや問題の調査を行う際は、仮説をテストするための最小限の実行可能な変更を行ってください。明示的に求められない限り、周囲のコードをリファクタリング、再構築、または「改善」しないでください。1つのタスクで2つ以上のファイルを修正する前に確認してください。
```

## 今後の展望

Claude Codeの使用状況は、レート制限と間違ったアプローチの摩擦によって深刻に損なわれている、Bash駆動でグラフィックスデバッグに特化したワークフローを明らかにしています。自律的なオーケストレーションパターンは両方のボトルネックを排除できます。

### 1. レート制限に強い自律タスクキューイング
**可能性**: セッションの半分以上がレート制限のために完全に失敗し、何時間もの潜在的な作業が蒸発しました。自律的なキューイングシステムは、作業を個別のシリアライズ可能なタスクグラフに事前計画できるため、容量が戻ったときにエージェントは人間の再プロンプトなしで正確に中断した場所から再開します。これにより、レート制限はセッションキラーから継続的なパイプラインの短い一時停止に変わります。

**始め方**: Claude CodeのTask/TaskCreateツールを使用して、ローカルのJSONまたはYAMLマニフェストに支えられた永続的なタスクキューを構築し、指数バックオフで再試行するBashラッパースクリプトと組み合わせます。

**Claude Codeに貼り付け**:
```
Claude Codeワークフロー用の自律タスクキューシステムを作成してください。.claude/task_queue.yamlというファイルを構築し、次のフィールドを持つ保留中のタスクのリストを保存します：id、description、status（pending/in_progress/done/blocked）、dependencies、retry_count、last_error。次に、.claude/run_queue.shというBashスクリプトを作成し、次のことを行います：1) キューから次の保留中のタスクを読み取る、2) そのタスクの説明でClaude Codeを呼び出す、3) レート制限エラーの場合、statusを'blocked'にタイムスタンプで設定し、指数バックオフでスリープしてから再試行、4) 成功時にタスクを完了とマークして次のタスクに移動、5) すべての試行を.claude/queue_log.mdに記録。最後に、'/queue add <description>'と言ってキューにタスクを追加できるSKILL.mdを作成してください。目標は、夜の作業をキューに入れて無人で実行し、レート制限の中断を乗り越えることです。
```

### 2. Metalシェーダーの並列エージェントデバッグ
**可能性**: あなたの最優先目標であるグラフィックスレンダリングのデバッグは、連続的な試行錯誤に苦しんでいました：間違ったシェーダーファイルの修正、間違ったdylibのデプロイ、テストサイクルの浪費。並列サブエージェントは、複数の仮説（シェーダーソース vs ビルドパイプライン vs ランタイムリンキング）を同時に調査し、時間のほんの一部で根本原因に収束できます。各エージェントは、報告する前に定義されたテストに対して独自の修正を検証します。

**始め方**: Claude CodeのTaskおよびTaskCreateツールを活用して並列サブエージェントを生成し、それぞれが特定の調査ブランチにスコープされ、調査結果をマージする調整親タスクを持ちます。

**Claude Codeに貼り付け**:
```
pob2macos Metalレンダリングプロジェクト用の並列デバッグワークフローをセットアップしてください。3つの並列サブタスクを生成して黄色テキストレンダリングバグを調査する親タスクを作成します：タスク1 - シェーダー分析：すべての.metalおよび.mmファイルで色出力割り当てをgrepし、黄色（RGB ~1,1,0）出力を生成する可能性のあるすべてのコードパスを特定し、正確なファイルと行番号をリストします。タスク2 - ビルドパイプライン監査：Makefile/ビルドスクリプトを読んで完全なビルドとデプロイパイプラインを追跡し、どの.dylibがランタイムディレクトリにコピーされるかを確認し、ランタイムでロードされているバイナリがコンパイルされたばかりのものと一致することを確認します（チェックサムを比較）。タスク3 - ランタイム状態検査：描画時にシェーダーに渡される実際の色値を出力する一時的なデバッグログをMetalバックエンドに追加し、ビルドして実行し、出力をキャプチャします。各サブタスクは調査結果を.claude/debug_reports/task_N.mdに書き込む必要があります。3つすべてが完了したら、レポートを単一の根本原因分析に統合し、提案された修正と、修正が機能することを確認するために実行できる検証テストを含めます。
```

### 3. 自己検証スキルインストールとテストパイプライン
**可能性**: スキルの作成とインストールセッションは、インストールされたスキルがシステムに認識されなかったときに摩擦に遭遇しました。これは検証が欠けている典型的なケースです。自律パイプラインは、スキルをインストールし、すぐに検出可能で実行可能であることをテストし、すべてのチェックに合格するまでインストールを反復できます。人間の介入なしで。このパターンは、「実際に機能したか？」が重要な質問である任意の構成またはセットアップタスクに一般化されます。

**始め方**: Claude Codeを使用して、スキルインストールをエンドツーエンドで検証するBashベースのテストハーネスを構築し、WriteおよびEditツールと組み合わせて見つかった問題を自動修正します。

**Claude Codeに貼り付け**:
```
Claude Codeセットアップ用の自己検証スキルインストールとテストシステムを構築してください。まず、現在の.claude/skills/ディレクトリを監査し、スキルを検出可能にする正確な形式、命名規則、ファイル構造を文書化します（SKILL.mdファイル、インデックスファイル、スキルの参照方法を確認）。次に、スキル名と説明を引数として取る.claude/install_skill.shというBashスクリプトを作成し、次のことを行います：1) 正しい形式でSKILL.mdを生成、2) 正しいディレクトリに書き込み、3) 次をチェックする検証スイートを実行：ファイルが正しい場所に存在する、YAML/Markdownフロントマターが有効、スキル名が一意で既存のスキルと競合しない、そして最も重要なことに — テストプロンプトでスキルを実際に呼び出して正しくトリガーされることを確認、4) チェックが失敗した場合、問題を自動的に診断して修正し、最大3回まで検証を再実行、5) テストレポートを.claude/skill_tests/<skill_name>_test.mdに出力。最近のgitコミットと未解決のTODOを要約する'/project-status'という単純なサンプルスキルをインストールすることで、このシステムをテストしてください。
```
